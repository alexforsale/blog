Branching dan Merging Sederhana

Mari kita lihat contoh sederhana dari branching dan merging dengan alur kerja yang kemungkinan besar kita alami di dunia nyata. Kita biasanya menggunakan langkah - langkah seperti ini:

  • Mengerjakan sesuatu pada sebuah website.
  • Membuat sebuah branch khusus untuk satu cerita yang sedang kita kerjakan.
  • Mengerjakan sesuatu dalam branch baru tersebut.

Ditahap ini, ternyata kita menerima telepon bahwa ada satu permasalahan(issue) yang butuh hotfix cepat. Kita lakukan seperti ini:

  • Pindah kembali ke branch production.
  • Buat branch baru untuk membuat hotfix-nya.
  • Setelah lolos test, merge branch hotfix tersebut dan push ke production.
  • Pindah kembali ke cerita awal yang kita sedang kerjakan sebelumnya.

Branching dasar

Pertama, anggaplah kita sedang mengerjakan sebuah project dan sudah memiliki beberapa commit di branch master.

Kita memutuskan untuk mengerjakan issue #53 dalam system tracking issue yang digunakan oleh perusahaan. Untuk membuat branch baru dan langsung berpindah ke branch tersebut kita menggunakan perintah

$ git checkout -b iss53
Switched to a new branch "iss53"

Ini merupakan perintah gabungan dari:

$ git branch iss53
$ git checkout iss53

Kita melanjutkan pekerjaan dan membuat commit baru. Yang membuat branch iss53 bergerak maju, karena kita checkout ke situ(disitu tempat yang dirujuk HEAD):

$ vim index.html
$ git commit -a -m 'added a new footer [issue 53]'

Sekarang kita mendapatkan telepon bahwa ada issue baru dengan website, yang harus dikerjakan secepatnya. Dengan git, kita tidak perlu merilis fix tersebut bersamaan dengan perubahan untuk iss53 yang baru kita buat, dan kita tidak perlu membalikkan perubahan - perubahan tersebut sebelum kita bisa mengerjakan fix di production. Yang perlu kita lakukan hanyalah pindah kembali ke branch master:

$ git checkout master
Switched to branch 'master'

Disini, working directory dari project kita sama seperti keadaan sebelum kita memulai issue #53, dan kita bisa fokus mengerjakan fix ini. Ini poin yang perlu diingat: ketika kita berpindah branch, git mereset working directory agar terlihat seperti saat commit terakhir di branch tersebut. Git akan menambah, mengurangi dan merubah file secara otomatis agar terlihat seperti itu.

Selanjutnya, kita mempunyai satu hotfix yang harus dikerjakan:

$ git checkout -b hotfix
Switched to a new branch 'hotfix'
$ vim index.html
$ git commit -a -m 'fixed the broken email address'
[hotfix 1fb7853] fixed the broken email address
 1 file changed, 2 insertions(+)

Kita lakukan beberapa kali test, untuk memastikan hotfix tersebut, dan akhirnya merge branch hotfix kembali kedalam branch master untuk dilepas ke production. Kita lakukan menggunakan perintah git merge:

$ git checkout master
$ git merge hotfix
Updating f42c576..3a0874c
Fast-forward
 index.html | 2 ++
 1 file changed, 2 insertions(+)

Perhatikan ada istilah “fast-forward” dalam merge tersebut. Karena commit C4 yang dipoint oleh branch hotfix yang kita merge berada persis didepan dari commit C2 lokasi kita sekarang, git hanya perlu menggerakan maju pointernya saja. Dengan kata lain, ketika kita mencoba merge satu commit dengan commit yang bisa dijangkau dengan mengikuti history commit pertama, git mempermudah dengan menggerakan maju pointernya karena tidak ada cabang yang perlu dimerge bersamaan - ini yang dinamakan “fast-forward”.

Perubahan kita sekarang berada didalam snapshot dari commit yang dipoint oleh branch master dan kita bisa merilis fix tersebut.

Setelah fix tersebut dirilis, kita siap untuk kembali mengerjakan issue yang tadi kita kerjakan sebelum terpotong oleh fix ini. Namun, sebelumnya kita perlu menghapus branch hotfix ini karena kita tidak lagi membutuhkannya - branch master sama - sama merujuk ke lokasi yang sama. Kita bisa menghapusnya dengan aman menggunakan tambahan -d dalam git branch

$ git branch -d hotfix
Deleted branch hotfix (3a0874c).

Sekarang kita bisa pindah kembali ke branch issue #53 dan melanjutkan pekerjaan.

$ git checkout iss53
Switched to branch "iss53"
$ vim index.html
$git commit -a -m 'finished the new footer [issue 53]'
[iss53 ad82d7a] finished the new footer [issue 53]
1 file changed, 1 insertion(+)

Perlu diingat bahwa semua yang baru kita kerjakan di branch hotfix tidak ada didalam branch iss53 ini. Jika kita perlu menariknya, kita bisa merge branch master kedalam iss53 dengan perintah git merge master, atau kita bisa menundanya dan merge branch iss53 ini ke branch master setelahnya.

Merging dasar

Anggaplah kita sudah memutuskan bahwa fix untuk issue #53 sudah selesai dan siap untuk di merge ke branch master. Untuk melakukan itu kita merge branch iss53 kedalam branch master sama seperti proses merging branch hotfix sebelumnya. Yang perlu kita lakukan adalah pindah ke branch yang akan menerima merge tersebut dan jalankan perintah git merge:

$ git checkout master
Switched to branch 'master'
$ git merge iss53
Merge made by the 'recursive' strategy.
index.html |    1 +
1 file changed, 1 insertion(+)

Terlihat berbeda dengan output dari merge branch hotfix sebelumnya. Ini karena, history bercabang di point awal. Commit di branch kita sekarang bukanlah ancestor langsung dari branch yang hendak kita merge, git perlu melakukan sesuatu terlebih dahulu sebelumnya. Yang dilakukan oleh git adalah merging tiga arah sederhana, menggunakan dua snapshot yang merujuk ke ujung branch dan ancestor serupa dari keduanya.

Daripada sekedar memajukan pointer branch kedepan, git membuat snapshot baru yang merupakan hasil dari merge tiga arah tadi dan otomatis membuat commit yang merujuk ke snapshot tersebut. Ini disebut merge commit, dan unik karena memiliki lebih dari satu parent.

Perlu diketahui pula bahwa git yang menentukan ancestor serupa yang akan dijadikan basis untuk merge-nya; ini merupakan satu perbedaan dibandingkan VCS lain seperti CVS atau Subversion(sebelum versi 1.5), dimana sang developer yang melakukan merging harus menentukan sendiri basis merge terbaik.

Sekarang setelah pekerjaan kita selesai dimerge, kita tidak lagi membutuhkan branch iss53 tadi, kita bisa close ticket dalam ticket-tracking system dan hapus branch-nya:

$ git branch -d iss53

Contoh Merge conflict

Terkadang, proses merging ini tidak berjalan mulus. Jika kita merubah satu bagian file yang sama dalam dua branch yang berbeda, git tidak akan mampu untuk melakukan merge secara bersih. Jika fix untuk issue #53 ternyata memodifikasi bagian dari file yang sama dengan branch hotfix, kita akan mendapatkan merge conflict seperti ini:

$ git merge iss53
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

Disini git belum membuat merge commit secara otomatis. Prosesnya dihentikan untuk sementara memberikan kita kesempatan untuk me-resolve conflict tersebut. Jika kita ingin melihat file mana yang tidak di merge setelah merge conflict kita bisa lihat melalui git status:

$ git status
On branch master
You have unmerged paths.
  (fix conflicts and run "git commit")

Unmerged paths:
  (use "git add <file>..." to mark resolution)

    both modified:      index.html

no changes added to commit (use "git add" and/or "git commit -a")

Semua yang memiliki merge conflict dan belum di resolve di-list sebagai unmerged. Git menambahkan penanda khusus didalam file - file tersebut, jadi bisa kita buka secara manual untuk me-resolve nya. File kita mungkin memiliki bagian seperti ini:

<<<<<<< HEAD:index.html
<div id="footer">contact : email.support@github.com</div>
=======
<div id="footer">
 please contact us at support@github.com
</div>
>>>>>>> iss53:index.html

Ini berarti versi yang di HEAD(branch master, karena disitulah kita checkout untuk melakukan merge) adalah bagian teratas dari block(semua yang diatas ======), sementara versi yang didalam branch iss53 adalah semua yang berada dibawahnya. Untuk resolve conflict ini kita harus memilih salah satu dari dua versi tersebut, atau membuat versi baru. Misalnya, kita bisa mengganti seluruh block tersebut dengan:

<div id="footer">
please contact us at email.support@github.com
</div>

Resolusi ini memiliki sedikit bagian dari masing - masing block tadi, dan tanda <<<<<<<, =======, dan >>>>>>> semuanya telah terhapus. Kita lakukan git add di setiap file yang unmerged untuk menandainya sebagai resolved. Staging file - file tersebut dalam kondisi ini berarti menandainya sebagai resolved dalam git.

Jika kita ingin menggunakan tool graphic untuk membantu resolve conflict seperti ini, kita bisa gunakan perintah git mergetool, yang nantinya akan membuka tool merge visual yang digunakan.

$ git mergetool

This message is displayed because 'merge.tool' is not configured.
See 'git mergetool --tool-help' or 'git help config' for more details.
'git mergetool' will now attempt to use one of the following tools:
opendiff kdiff3 tkdiff xxdiff meld tortoisemerge gvimdiff diffuse diffmerge ecmerge p4merge araxis bc3 codecompare vimdiff emerge
Merging:
index.html

Normal merge conflict for 'index.html':
  {local}: modified file
  {remote}: modified file
Hit return to start merge resolution tool (opendiff):

Jika kita ingin menggunakan mergetool lain selain tool default(git menggunakan opendiff dalam contoh ini karena perintah tersebut dijalankan didalam Mac), kita dapat melihat tool - tool lain yang didukung, cukup ketik nama tool tersebut.

Setelah kita keluar dari mergetool tadi, git akan bertanya apakah proses merge nya sukses. Jika kita menulis demikian maka file - file tersebut akan ditandai sebagai resolved. Kita bisa menjalankan kembali git status untuk memastikan benar sudah diresolved.

$ git status
On branch master
All conflicts fixed but you are still merging.
  (use "git commit" to conclude merge)

Changes to be committed:

    modified:   index.html

Jika kita puas dengan hasil tersebut dan memastikan semua conflict sudah di staged, kita jalankan git commit untuk menyudahi proses merge-nya. Secara default commit message-nya seperti ini:

Merge branch 'iss53'

Conflicts:
    index.html
#
# It looks like you may be committing a merge.
# If this is not correct, please remove the file
#	.git/MERGE_HEAD
# and try again.


# Please enter the commit message for your changes. Lines starting
# with '#' will be ignored, and an empty message aborts the commit.
# On branch master
# All conflicts fixed but you are still merging.
#
# Changes to be committed:
#	modified:   index.html
#

Sebaiknya kita tambahkan detail lebih mendalam lagi mengenai proses merging-nya dalam commit message.
